# VoidManager `.db` — Real Schema Findings

Discovered by inspecting a real DB (`scripts/inspect-db.mjs`). **This overrides the plan's
assumption of simple `Project/Building/Story/Void` tables with literal columns.** The DB is a
**dictionary-encoded, append-only, mutation-versioned object store** produced by
VoidManagerCore's `DatabaseObject` / `MutableObject` engine. ~556 tables.

## Three table families

1. **`D_<Type>` dictionary tables** — `(ID INTEGER PK AUTOINCREMENT, Value <typed> UNIQUE)`.
   Map small int IDs → literal values. The literal value lives here, NOT in the entity row.
   - `D_Boolean` — **`1 → true`, `2 → false`** (Value is a `bit`).
   - `D_StatusOfApproval` — `1="open for review"`, `2="approved"`, `3=…` (only the statuses ever
     used are present; the full six-value list is defined app-side in `schema.ts`).
   - `D_SequenceName` — `1="AT-01-10"`, `2="WA-01-10"`.
   - `D_Category` — `"Walls"`, `"category"`, … (host element category).
   - `D_EmailAddress` — user emails. `D_Name`, `D_Model` — host element name/model.
   - `D_GUID` — **GUIDs *and* Revit ElementId integers-as-strings** both live here
     (`"ec9e90e7-…"` and `"673660"`). Confirms the plan's ExternalId ambiguity pitfall.
   - `D_NumericOneDecimal`, `D_NumericOneDecimalUnsigned`, `D_NumericInteger` — numbers (mm, etc.).
   - `D_TimestampUtc` — datetimes. `D_ID` — 28-byte hash object identities. `D_Notes`, `D_Message`,
     `D_Severity`, `D_CauseType`, `D_TimeSpent`, `D_PropertyName`.

2. **Entity tables** — `VoidCircle`, `VoidRectangle`, `Story`, `User`, `Building`, `Project`,
   `HostElementSuper`, `HostElementSubCircle`, `HostElementSubRectangle`, `Point3D`, `Vector3D`,
   `PointOfAttention`, … **Every non-PK column is an INT foreign key** into a `D_` table or another
   entity table. No literals are stored inline.

3. **Snapshot / staging copies** — `*_temp` tables and deeply nested
   `Project_Buildings_Stories_VoidCircle_*` flattened trees (per-project "pull" snapshots, plus
   `_Mutations_`, `_AssignedTo_`, `_CreatedBy_`, `_Reviewers_`, `_PointsOfAttention_` siblings).

## VoidCircle / VoidRectangle columns (all INT refs)

`ID` (PK) · `HostId`→Story · `IsValid`→D_Boolean · `CreatedOn`→D_TimestampUtc ·
`CreatedBy`→User · `IsClosed`→D_Boolean · `Notes`→D_Notes · `Thickness`→D_Numeric* ·
`ExternalId`→D_GUID · `AssignedTo`→User · `StatusOfApproval`→D_StatusOfApproval ·
`Location`→Point3D · `Direction`→Vector3D · `SequenceName`→D_SequenceName ·
`MergedIntoVoidId`→VoidCircle.ID · `IsAutogenerated`→D_Boolean · `Diameter`,`Tolerance`→D_Numeric*
(rectangle adds `Width`,`Height`,`ToleranceAbove/Right/Below/Left`→D_Numeric*).

`Point3D` = `(ID, X, Y, Z)` where X/Y/Z are themselves INT refs into numeric dictionaries
(Revit internal mm). `Vector3D` likewise.

## Versioning (critical for reads)

There are **multiple rows per logical void** (same `ExternalId`) = mutation history. `IsValid`
(→D_Boolean) flags the current version; superseded versions resolve to `false`. Two sampled
`VoidCircle` rows shared `ExternalId=1` but differed in `StatusOfApproval` (2 vs 1) and
`Diameter` — different versions of one void.

**Read recipe (current voids):** join each ref column through its dictionary/entity table and
filter to the current version, e.g.

```sql
SELECT v.ID,
       sa.Value  AS status,
       sq.Value  AS sequenceName,
       g.Value   AS externalId,
       dia.Value AS diameter
FROM VoidCircle v
JOIN D_Boolean valid          ON valid.ID = v.IsValid AND valid.Value = 1   -- current version only
JOIN D_StatusOfApproval sa    ON sa.ID    = v.StatusOfApproval
LEFT JOIN D_SequenceName sq   ON sq.ID    = v.SequenceName
LEFT JOIN D_GUID g            ON g.ID     = v.ExternalId
LEFT JOIN D_NumericOneDecimalUnsigned dia ON dia.ID = v.Diameter
-- + Point3D join (and its X/Y/Z numeric joins) for Location, User join for AssignedTo, etc.
```

## Write-back risk (the hard part — NOT a simple UPDATE)

Changing a status is **not** `UPDATE VoidCircle SET StatusOfApproval=2 WHERE ID=…`. The faithful
procedure (as performed by `VoidManagerCore.PushToDatabase`, which lives in the DLL) is some
combination of:

1. Find or `INSERT` a `D_StatusOfApproval` row for the target string → get its ID.
2. Respect the mutation/versioning model: likely insert a **new void version row** (new PK) with the
   new `StatusOfApproval` ref, flip `IsValid` (new=true, prior=false), add a `*_Mutations_` entry,
   and set `CreatedOn`/`CreatedBy`. Possibly also update the `_temp`/snapshot tables.

A naive in-place UPDATE may not be recognized by the Revit add-in or may corrupt history.
**Any write path must be validated** by writing, then re-opening the DB in VmoViewer / the Revit
add-in to confirm the change is honored.

## Open questions before building the write path

- Exact mutation/version write procedure (reverse-engineer `*_Mutations_` tables, or drive the DLL).
- Which tables the add-in reads on pull — base entity tables vs `_temp` snapshots.
- Void → Project scoping (the `HostId` → Story → Building → Project chain) for the project selector.

## Implication

**Reads are feasible** in-browser via sql.js (dictionary-decoding joins + current-version filter).
**Write-back is the open architectural question** — see the strategy options raised with the user.
